# 기술 부록 — 살래말래 (Live or Leave)

> README.md가 다루지 않는 설계 원칙, 페르소나 데이터 구축 방법론, 알려진 한계,
> 재현 방법을 정리한 기술 부록입니다. (개발 과정의 세션별 변경 이력·트러블슈팅
> 일지는 최종본에서는 제외했습니다.)

## 절대 바꾸면 안 되는 설계 원칙 (이유 포함)

1. **LLM은 입구·출구에만. 계산은 전부 백엔드.**
   정규화·스코어링·정렬은 `scoring.py`가 결정론적으로 한다. LLM에게 산수를
   시키지 않는다. 이유: 재현성·속도·정확성. "에이전트니까 LLM이 다 해야"는 오해다.

2. **LLM은 숫자를 만들지 않고 라벨만 고른다.**
   `parse_intent`는 카테고리별 중요도를 4단계 라벨(Importance)로만 출력.
   라벨→가중치(0.0~1.0) 변환은 `scoring.preference_to_weights`가 결정론적으로.
   이유: LLM이 매번 "0.8" 같은 숫자를 즉흥적으로 뽑으면 랭킹이 흔들린다.

3. **출구 LLM은 제공된 수치만 근거로 설명.**
   추천 행정동의 실제 지표를 프롬프트에 넣고, "수치에 없는 내용은 추측 금지"를
   강제한다. 이유: 할루시네이션 방지. 요구 미충족도 솔직히 명시한다.

4. **분석 단위는 행정동(424개). 구 단위 아님.**
   구 단위는 "강남구 전체 편의점 805개"로 뭉뚱그려 대치동·개포동을 구분 못 함.
   행정동으로 내려 같은 구 안 변별력을 확보했다. (MAUP 문제 해결)

5. **지표별 처리 방식이 다르다 (데이터 해상도를 정직하게 반영).**
   - 반경 1km 내 개수형: 편의점·마트·병원·버스·CCTV·공원 (인구로 나눠 밀도화)
   - 최근접 거리형: 지하철 (거리감쇠 exp(-d/500))
   - 구 상속: 범죄율 (행정동 범죄 데이터가 공개 안 됨 → 소속 구 값 물려받음)
   - 행정동 고유: 생활인구
   범죄를 억지로 행정동화하지 않는다. 없는 데이터를 지어내지 않는 게 원칙.

6. **분모는 생활인구(주민등록 아님).**
   LOCAL_PEOPLE 데이터를 씀. 유동인구 포함이라 "실제 그 지역을 이용하는 사람
   대비"로 평가됨.

7. **출구 LLM 문장에는 가중치·기여도·raw 지표 개수를 노출하지 않는다.**
   `scoring.py`/`tools.py`가 계산한 가중치·기여도·raw 수치(범죄율, CCTV·공원
   개수 등)는 `explain`이 "판단"할 근거로는 계속 쓰지만, 사용자에게 보여줄
   문장에는 그 숫자를 그대로 인용하지 않고 자연어로 풀어 쓴다("mobility 가중치
   0.526" 대신 "대중교통 접근성이 좋아 이동 부담이 적습니다"). 사용자가 직접
   요청한 업종의 실제 개수(예: "헬스장 3곳")는 예외 — 이건 내부 스코어링
   지표가 아니라 사용자가 물어본 사실이라 그대로 알려준다. raw 수치 자체는
   지도 hover 툴팁과 "🔍 적용된 필터 검증" expander(`streamlit_app.py`)에
   여전히 노출되므로 완전히 숨겨지는 건 아니고, "메인 응답 문장"에서만
   걷어낸 것이다.

8. **도구를 남발하지 않는다.**
   흐름은 입구→백엔드→출구로 고정. LLM이 여러 도구를 자율 연쇄 호출하는
   구조는 의도적으로 배제(오버엔지니어링). 되묻기 1회만 agentic 분기.

## 페르소나 시나리오 설계 기준 (발표 검증용)

멘토링 피드백의 핵심은 "서비스가 기존 서비스와 어떻게 다르고, 실제 사용자 요구를
현재 데이터 스키마가 얼마나 커버하는지"를 보여주는 것이다. 그래서 시나리오 30개는
임의 예시가 아니라, `QUTUMENT/nemotron-personas-korea-extended` 샘플에서 뽑은
합성 페르소나를 바탕으로 만든 **대표 사용자 요구 검증셋**이다. 최종 결과물은
[require.md](require.md)에 정리돼 있다.

> **참고**: 중간 산출물(페르소나 원본 샘플 CSV, 후보 질문 84개 등)은 용량 문제로
> `.gitignore` 처리돼 있다(원본 기준 약 23MB). 저장소에 실제로 포함된 건 최종
> 결과물인 `require.md`와, 원본 데이터로부터 그 결과물을 재생성하는 스크립트
> (`scripts/sample_qutument_personas.py`, `scripts/build_persona_scenarios.py`)
> 2개다.

### 데이터 출처 및 샘플링 방법

- 원본: `QUTUMENT/nemotron-personas-korea-extended` — NVIDIA `Nemotron-Personas-Korea`
  기반 확장판 미러. CC BY 4.0. 합성 페르소나이며 실제 개인 데이터가 아니다.
- 원본 데이터셋은 51개 컬럼, 1,000,000행. 시나리오 생성에 필요한 36개 컬럼만
  읽고, 분석용 파생 컬럼 5개(`age_group`, `occupation_group`, `housing_group`,
  `income_group`, `is_seoulish`)를 추가했다.
- 샘플링은 단순 랜덤이 아니라 **서울 필터 + 재현 가능한 셔플(seed=42) + 약한
  층화 샘플링**이다. `district`에 "서울-"이 들어간 행만 필터링하고, `age_group`/
  `family_type`/`occupation_group`/`housing_group`/`income_group` 조합을 층화
  키로 사용해 특정 조합이 샘플을 독점하지 않게 했다.
- 페르소나 텍스트(직업·가족·건강·취미 등)에서 주거 추천 신호(운동/반려동물/
  프리랜서/이동성/야간안전/병원/부모돌봄/조용함/월세부담/편의/자녀학교 등
  11종)를 태깅해, 3,000개 서울 샘플 중 신호가 강한 500개를 추렸다.

### 84개 후보 → 최종 30개 선별 흐름

1. 서울 3,000개 샘플에 주거 추천 신호를 태깅·점수화 → 상위 500개 추출.
2. 라이프스타일/부족 데이터/되묻기 유형 템플릿으로 질문 후보 84개 생성(원본
   페르소나의 연령대·직업·거주 구·가족 형태를 반영해 실제 사용자 맥락처럼 구성).
3. 각 질문에 필요한 데이터 필드·현재 서비스 카테고리를 매핑.
4. 각 질문을 `answerable`/`partial`/`not_answerable`/`clarify`로 태깅.
5. 발표 검증 목적에 맞춰 최종 30개를 비율대로 선별: `answerable` 18개,
   `partial` 8개, `not_answerable` 1개, `clarify` 3개.

핵심은 "데이터셋에서 자동으로 떨어진 질문 30개"가 아니라 **데이터셋 근거가 있는
질문 후보를 만들고, 현재 서비스의 데이터 커버리지 검증 목적에 맞게 큐레이션한
30개**라는 점이다.

### 시나리오 우선순위 기준

단순 "평균적인 이사 수요자"가 아니라, **집값에는 잘 반영되지 않지만 자기 생활에서
중요한 세부 조건이 뚜렷한 사람들**을 우선했다.

1. 라이프스타일이 확고한 MZ/1인 직장인 계열(러너, 헬스장 이용자, 반려동물
   양육자, 프리랜서/창작자, 이동성 중시자 등)
2. 1인 가구가 아니어도 세부 주거 요구가 선명한 사람(채광, 방음, 밤길 안정감,
   병원 접근성 중시 등)
3. 발표 데모에 쓰기 좋은 생활 편의형(병원 접근성, 헬스장/공원, 상권 편의시설 등)

### 부족 데이터(현재 스키마로 답하기 어려운 요구) 처리 방침

월세·전세 시세, 실제 통근시간, 소음/방음, 반려동물 인프라, 남향·채광은 현재
행정동 공공지표로 직접 검증하기 어렵다. 이런 요구는 추천을 억지로 꾸미지 않고,
"현재 데이터로 확인할 수 없다"는 한계를 설명에 명시하는 방향으로 처리했다
(`app/agent/unsupported_requirements.py`).

### 재생성 방법

```bash
.venv/bin/python scripts/sample_qutument_personas.py    # 원본 HF 데이터셋 → 서울 중심 샘플
.venv/bin/python scripts/build_persona_scenarios.py      # 샘플 → 후보 질문 → 최종 30개(require.md 기준 산출물)
```

## 알려진 한계 / 주의

- 범죄는 구 단위 → 같은 구 안 행정동은 범죄율 동일 (데이터 한계, 정직한 처리).
- 행정동 크기 중앙값 0.97km²(반지름 ~557m)라 반경 1km면 대부분 커버되나,
  거대 행정동(최대 12.7km²)은 중심점 근사 오차가 큼. Kakao "근처" 필터의
  `NEAR_RADIUS_KM`도 동 중심점 기준이라 같은 종류의 근사 오차를 가진다.
- **기관마다 행정구역 데이터 버전이 다르다.** 서울시 자체 시스템(상권분석서비스,
  생활인구)과 통계청 SGIS는 행정구역 개편 반영 시점이 다르다 — 예: 강남구
  일원2동은 2023년에 개포3동으로 편입됐는데, 서울시 쪽 데이터는 아직도 일원2동을
  별개로 취급한다. `dong_metrics.csv`는 서울시 쪽(구 경계) 기준이고,
  `dong_boundaries.geojson`은 통계청 SGIS(신 경계) 기준이라 지도 생성 시
  이름 매핑 보정이 필요했다(`build_dong_boundaries.py`의 `NAME_FIXES` 참고).
- **지도 카메라 위치(확대/이동)가 위젯 조작 시 초기화된다.** Plotly가 지도
  내용(색·핀)이 바뀔 때마다 컴포넌트를 완전히 새로 마운트하는 구조적 한계로
  파악해 현재는 받아들이고 있다.
- Streamlit 풀스크린 CSS(`streamlit_app.py`의 `FULLSCREEN_CSS`)가 `data-testid`
  등 Streamlit 비공식 내부 구현에 의존한다 — Streamlit 버전을 올리면 레이아웃이
  깨질 수 있다.
- **Kakao 캐시 디렉토리(`DEFAULT_CACHE_DIR`)는 `dataset/` 밑이 아니라 프로젝트
  루트의 `.kakao_cache/`에 둔다.** `docker-compose.yml`이 `dataset/`을 읽기
  전용(`:ro`)으로 마운트하므로, 캐시를 그 아래 두면 컨테이너에서 쓰기 실패
  (`OSError: Read-only file system`)로 죽는다(GCE 배포에서 실제로 겪음).
  `dataset/` 아래에 쓰기가 필요한 파일을 새로 추가할 땐 항상 이 제약을 의심할 것.
- **GCE VM의 `.env`는 로컬 `.env`와 별개 파일이라 자동 동기화되지 않는다.**
  CI/CD(`git pull` + `docker compose up -d --build`)는 코드만 갱신하고
  `.env`(gitignore 대상)는 건드리지 않는다. 로컬 `.env`에 새 키를 추가해도
  VM에는 반영되지 않으므로, VM 쪽 `.env`도 직접 SSH로 갱신해야 한다(예:
  `gcloud compute ssh live-or-leave-vm --zone=asia-northeast3-a`). 갱신 후엔
  `docker compose up -d --force-recreate`로 컨테이너를 재생성해야 새 환경변수를
  읽는다(단순 `restart`는 기존 컨테이너에 이미 구워진 환경변수를 그대로 씀).

## 재현 방법

```bash
pip install -r requirements.txt   # Python 3.9+ 필요 (LiteLLM 의존성)
python demo.py                    # 시나리오 데모 (실제 Solar API, .env 필요)
uvicorn main:app --reload         # FastAPI 서버 (http://127.0.0.1:8000)
streamlit run streamlit_app.py    # 지도 UI (http://localhost:8501) — 키 없으면 자동 mock 폴백
python -m pytest tests/           # 전체 유닛테스트 (181개, MockLLM 기반, 키 없이도 실행됨)
```

`dong_metrics.csv`/`dong_boundaries.geojson`은 이미 커밋돼 있어 원본 공공데이터
없이도 바로 실행된다. `streamlit_app.py`는 `.env`에 `UPSTAGE_API_KEY`가 없으면
자동으로 MockLLM으로 폴백해 화면에 안내 문구를 띄우므로, 키 없이도 지도 UI를
바로 확인할 수 있다.

SSE 스트리밍 직접 확인:

```bash
curl -N --get "http://127.0.0.1:8000/recommend" \
  --data-urlencode "text=야근이 잦고 차가 없어서 지하철이 중요해. 밤에 안전한 동네였으면 좋겠어."
```

Docker Compose 실행:

```bash
cp .env.example .env   # .env에 UPSTAGE_API_KEY 입력
docker compose up -d --build
curl http://127.0.0.1:8000/health
```
