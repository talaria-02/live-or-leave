# Mock Policy

## 1. 목적

오늘 MVP는 전체 데이터를 완벽히 정제하는 것이 아니라, 다음 핵심 시나리오를 자치구 단위로 성공시키는 것이 목표다.

> 나는 러닝을 좋아하는 20대 남자야. 근처에 공원과 햄버거집이 많은 동네를 추천해줘.

따라서 실제 데이터로 바로 생성 가능한 feature는 실제값으로 사용하고, 현재 보유 데이터만으로 직접 계산하기 어려운 feature는 현실성 있는 Mock 또는 파생값으로 보완한다.

## 2. Mock 또는 파생 처리한 컬럼

| 컬럼 | 처리 방식 | 이유 | 생성 기준 | 추후 교체 가능한 실제 데이터 |
|---|---|---|---|---|
| `park_count` | Mock-derived | 보유 공원 데이터는 자치구별 면적/비율 통계이며 개별 공원 목록이 없다. | `park_area_per_person`, `park_ratio`를 min-max 정규화한 공원 기반 점수로 12~44 범위 count 생성 | 서울시 공원 위치/목록 데이터, 공원명/자치구/면적 컬럼 |
| `large_park_count` | Mock-derived | 개별 공원 면적 단위 데이터가 없어 대형 공원 수를 직접 산출할 수 없다. | 공원 기반 점수와 주요 대형 녹지 이미지가 강한 자치구 보정값을 결합 | 개별 공원 목록 + 면적 컬럼으로 threshold 이상 공원 수 집계 |
| `running_friendly_score` | Mock-derived score | 러닝 친화도는 산책로, 보행환경, 조도, 야간 안전성 등 추가 데이터가 필요하다. | `park_count`, `park_area_per_person`, `park_ratio`, `large_park_count`의 가중합을 0~100 점수화 | 산책로, 체육시설, 공원 CCTV, 조도, 보행환경 데이터 |
| `commercial_area_score` | Derived aggregate score | 생활 인프라 밀집도는 단일 원천 컬럼이 아니라 여러 상권 feature의 종합 점수다. | `food_count`, `cafe_count`, `hamburger_count`, `fastfood_count`의 min-max 정규화 가중합 | 상권 업종별 세분화, 편의점/문화시설/헬스장/대학가 데이터 |

## 3. 실제 데이터에서 생성한 컬럼

| 컬럼 | 원천 데이터 | 생성 기준 |
|---|---|---|
| `park_area_per_person` | `공원(1인당+공원면적)_20260708231051.xlsx` | 자치구별 1인당 공원면적 |
| `park_ratio` | `공원(공원율)_20260708231738.xlsx` | 자치구별 공원율 |
| `food_count` | `소상공인시장진흥공단_상가(상권)정보_서울_202603.csv` | `상권업종대분류명=음식`인 업소 수 |
| `cafe_count` | 상가/상권 데이터 | 음식 업종 중 카페/커피/디저트/베이커리/브랜드 키워드 매칭 |
| `hamburger_count` | 상가/상권 데이터 | 음식 업종 중 `상호명`, `상권업종소분류명` 기준 햄버거/버거/주요 브랜드 키워드 매칭 |
| `fastfood_count` | 상가/상권 데이터 | 음식 업종 중 `상호명`, `상권업종소분류명`, `표준산업분류명` 기준 패스트푸드/햄버거/치킨/피자/주요 브랜드 키워드 매칭 |

## 4. 주의사항

- 상호명 기반 키워드 매칭은 오탐 가능성이 있다.
- `hamburger_count`는 표준산업분류명의 `피자; 햄버거; 샌드위치 및 유사 음식점업`이 피자/샌드위치 업소까지 과하게 포함하는 문제가 있어, 오늘은 상호명과 소분류명 중심으로 좁혀 계산한다.
- 오늘은 MVP 데모가 목적이므로 업종 컬럼을 우선 사용하고, 부족한 경우 상호명 키워드를 보조로 사용한다.
- Mock-derived 컬럼은 실제값처럼 추천 로직에 입력되지만, 발표와 문서에서는 Mock 또는 파생값임을 명시한다.
- 추천 로직은 Mock 여부를 분기 조건으로 사용하지 않는다.
- 추후 실제 데이터가 확보되면 `processed/region_features.csv`의 컬럼 구조는 유지하고 값만 교체한다.

## 5. 교체 우선순위

1. 개별 공원 목록 데이터 확보 후 `park_count`, `large_park_count` 실제값 전환
2. 산책로/체육시설/보행환경 데이터 확보 후 `running_friendly_score` 실제 점수화
3. 상권 업종 세분화 및 영업 상태 반영 후 `commercial_area_score` 고도화
4. 좌표 기반 거리 계산이 가능해지면 행정동/지도 히트맵으로 확장
