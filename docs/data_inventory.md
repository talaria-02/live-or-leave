## 1. MVP 핵심 시나리오

오늘 MVP의 핵심 사용자 질의는 다음이다.

> 나는 러닝을 좋아하는 20대 남자야. 근처에 공원과 햄버거집이 많은 동네를 추천해줘.

이 질의에서 추출해야 할 주요 의도는 다음과 같다.

- 러닝을 좋아함: 공원, 산책로, 녹지, 운동하기 좋은 환경이 중요하다.
- 20대 남자: 음식점, 카페, 활동성 높은 상권 등 생활 인프라 선호를 보조적으로 고려할 수 있다.
- 공원 많음: `park_count`, `park_area_per_person`, `park_ratio` 등 공원 관련 feature가 필요하다.
- 햄버거집 많음: `hamburger_count`, `fastfood_count` 등 상권/음식점 feature가 필요하다.
- 동네 추천: 오늘 MVP에서는 서울 25개 자치구 단위로 추천한다.

## 2. 추천 단위

- 오늘 추천 단위는 **서울 25개 자치구**로 제한한다.
- 행정동 단위 추천, 좌표 기반 거리 계산, 지도 히트맵은 오늘 범위에서 제외하거나 보류한다.
- 좌표 데이터가 있는 경우에도 오늘은 공간 조인보다 자치구 컬럼이 있는 데이터를 우선 사용한다.

## 4. 오늘 사용할 데이터

오늘 실제 데이터로 우선 사용할 데이터는 다음 3개다.

1. `공원(1인당+공원면적)_20260708231051.xlsx`
   - 핵심 이유: 자치구별 공원 면적과 1인당 공원면적이 이미 있어 자치구 추천 단위와 잘 맞는다.
   - 만들 feature:
     - `park_area`
     - `park_area_per_person`
     - `walkable_park_area`
   - 사용 판단:
     - 오늘 핵심 시나리오의 "러닝", "공원 많음" 조건을 설명하기 위한 핵심 데이터로 사용한다.

2. `공원(공원율)_20260708231738.xlsx`
   - 핵심 이유: 자치구별 공원율이 있어 전체 지역 면적 대비 공원 비중을 계산하지 않고 바로 활용할 수 있다.
   - 만들 feature:
     - `park_ratio`
     - `park_area`
   - 사용 판단:
     - `park_score` 산정의 핵심 입력값으로 사용한다.

3. `소상공인시장진흥공단_상가(상권)정보_서울_202603.csv`
   - 핵심 이유: `시군구명`, `상호명`, 업종 분류가 있어 자치구별 음식점/카페/햄버거집 집계가 가능하다.
   - 만들 feature:
     - `food_count`
     - `restaurant_count`
     - `cafe_count`
     - `hamburger_count`
     - `fastfood_count`
     - `commercial_area_score`
   - 햄버거집 판별 기준:
     - `상호명`, `상권업종소분류명`, `표준산업분류명`에 아래 키워드가 포함되는 경우 후보로 본다.
     - 키워드: `햄버거`, `버거`, `롯데리아`, `맥도날드`, `버거킹`, `맘스터치`, `KFC`, `프랭크버거`, `노브랜드버거`, `패스트푸드`
   - 사용 판단:
     - 오늘은 완벽한 브랜드 분류보다 핵심 시나리오 데모가 목적이므로 키워드 기반 집계로 충분하다.

|---|---|---|---|
| `running_friendly_score` | 러닝 친화도는 공원 면적만으로 충분하지 않고 산책로, 경사, 보행 환경, 야간 안전성 등이 필요하다. | `park_ratio`, `park_area_per_person`, `large_park_count` 후보를 바탕으로 단순 Mock 점수 사용 | 공원 위치, 산책로, 체육시설, CCTV, 조도 데이터 결합 |
| `large_park_count` | 현재 공원 통계는 면적 총량 중심이며 개별 공원 단위 대형 공원 수가 바로 확인되지 않는다. | 오늘은 Mock 또는 보류 | 개별 공원 목록 데이터 확보 후 자치구별 대형 공원 수 집계 |
| `large_park_access_score` | 접근성은 사용자 위치 또는 자치구 중심점과 공원 좌표 간 거리 계산이 필요하다. | 오늘 제외 | 좌표 기반 공간 조인 및 거리 계산 |
| `walkability_score` | 보행 친화도는 도로, 보행로, 산책로, 교통안전 데이터가 필요하다. | 오늘 제외 | 산책로/보행환경 데이터 추가 수집 |

## 6. 오늘 보류할 데이터

오늘 핵심 시나리오에는 공원과 햄버거집이 직접 조건으로 들어간다. 따라서 아래 데이터는 보류한다.

- 전월세 데이터
  - 이유: 이번 질의에는 예산, 월세, 전세 조건이 없다.
  - 추후 활용: `rent_score`, `avg_monthly_rent`, `median_deposit`

- 지하철/버스 데이터
  - 이유: 이번 질의에는 교통 조건이 없다.
  - 추후 활용: `subway_station_count`, `bus_stop_count`, `transit_score`

- 범죄/CCTV 데이터
  - 이유: 안전성은 러닝과 간접적으로 연결될 수 있지만 오늘 핵심 조건은 아니다.
  - 추후 활용: `safety_score`, `night_running_safety_score`, `park_cctv_count`

- 금연구역 데이터
  - 이유: 쾌적성 지표 후보지만 오늘 사용자 질의와 직접 관련이 낮다.
  - 추후 활용: `non_smoking_zone_count`, `clean_environment_score`

- 어린이보호구역 데이터
  - 이유: 20대 1인/활동성 사용자 시나리오와 직접 관련이 낮다.
  - 추후 활용: 가족 친화형 주거 추천, 교통안전 점수

## 7. 핵심 시나리오용 Feature 후보

오늘 핵심 시나리오를 위해 만들 feature 후보는 다음과 같다.

| feature | 사용 여부 | source | 설명 |
|---|---|---|---|
| `region_name` | 사용 | 모든 자치구 단위 데이터 | 서울 25개 자치구명 |
| `park_score_source` | 사용 | 공원 면적/공원율 데이터 | `park_area_per_person`, `park_ratio` 기반 공원 점수 출처 |
| `hamburger_score_source` | 사용 | 상가/상권 데이터 | 상호명/업종명 키워드 기반 햄버거집 점수 출처 |
| `park_count` | Mock 대체 | 개별 공원 목록 필요 | 현재 보유 공원 통계만으로는 직접 계산 어려움 |
| `park_area_per_person` | 사용 | `공원(1인당+공원면적)` | 자치구별 1인당 공원면적 |
| `park_ratio` | 사용 | `공원(공원율)` | 자치구별 행정구역면적 대비 공원면적 비율 |
| `food_count` | 사용 | 상가/상권 데이터 | 자치구별 음식 업종 수 |
| `hamburger_count` | 사용 | 상가/상권 데이터 | 자치구별 햄버거/버거/패스트푸드 키워드 매칭 업소 수 |
| `cafe_count` | 사용 | 상가/상권 데이터 | 자치구별 카페 업종 수 |
| `running_friendly_score` | Mock 대체 | 공원 통계 + 추후 산책로/안전 데이터 | 오늘은 공원 관련 지표 기반 Mock 점수로 보완 |
| `large_park_count` | Mock 대체 | 개별 공원 목록 필요 | 오늘은 직접 계산하지 않음 |
| `commercial_area_score` | 사용 후보 | 상가/상권 데이터 | 음식점, 카페, 생활 인프라 집계 기반 보조 점수 |

오늘 최소 feature set은 다음으로 정의한다.

- `region_name`
- `park_score_source`
- `hamburger_score_source`
- `park_area_per_person`
- `park_ratio`
- `food_count`
- `hamburger_count`
- `cafe_count`
- `running_friendly_score`는 Mock
- `park_count`, `large_park_count`는 Mock 또는 추후 확장

## 8. 추후 데이터 확장 방향

- 행정동 단위 추천
  - 상가/상권 데이터는 행정동 컬럼이 있으므로 행정동 추천으로 확장하기 좋다.
  - 공원 데이터도 개별 공원 위치 또는 행정동별 공원 통계를 추가 확보해야 한다.

- 좌표 기반 접근성
  - 상가/상권, 지하철, 버스, CCTV 데이터는 좌표가 있으므로 공간 조인과 거리 계산으로 확장 가능하다.
  - 오늘은 자치구 집계만 사용한다.

- 러닝 친화도 고도화
  - 공원 면적, 공원율만으로는 러닝 친화도를 충분히 설명하기 어렵다.
  - 산책로, 체육시설, 야간 안전성, 공원 CCTV, 조도 데이터 등을 결합하면 `running_friendly_score`를 실제값으로 만들 수 있다.

- 주거비/교통 조건 추가
  - 전월세 데이터는 예산 조건이 들어오는 시나리오에서 중요하다.
  - 지하철/버스 데이터는 출퇴근, 역세권 조건이 들어오는 시나리오에서 중요하다.

- 안전성 조건 추가
  - 범죄/CCTV 데이터는 야간 러닝, 안전한 동네, 치안 조건이 들어오는 경우 보조 지표로 사용할 수 있다.
  - 단, 범죄 데이터는 해석 민감도가 높으므로 추천 사유 표현에 주의해야 한다.

## 9. 오늘의 결론

오늘은 전체 데이터 정제가 아니라, 공원과 햄버거집 중심의 핵심 시나리오를 성공시키는 것이 목표다.

추천 단위는 서울 25개 자치구로 제한한다.

실제 데이터로 만들 수 있는 feature는 실제값으로 사용하고, 러닝 친화도처럼 직접 계산이 어려운 feature는 현실성 있는 Mock으로 보완한다.

주거비, 교통, 안전 데이터는 이번 시나리오에서는 보류하거나 추후 확장용으로 관리한다.

따라서 오늘 MVP 데이터 전략은 다음과 같다.

- 공원 관련 feature는 자치구별 공원 면적/1인당 공원면적/공원율 데이터를 사용한다.
- 햄버거집 및 생활 인프라 feature는 상가/상권 데이터에서 자치구별로 집계한다.
- `running_friendly_score`, `large_park_count`처럼 추가 데이터나 공간 계산이 필요한 지표는 Mock 또는 추후 확장으로 둔다.
- 전월세, 교통, 범죄/CCTV, 금연구역, 어린이보호구역 데이터는 오늘 핵심 시나리오에는 넣지 않는다.
