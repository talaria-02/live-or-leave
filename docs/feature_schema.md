# Feature Schema

## 1. MVP 핵심 시나리오

오늘 MVP의 핵심 사용자 질의는 다음이다.

> 나는 러닝을 좋아하는 20대 남자야. 근처에 공원과 햄버거집이 많은 동네를 추천해줘.

이 시나리오에서 추천 로직이 반영해야 할 조건은 다음과 같다.

- 러닝을 좋아함: 공원, 녹지, 산책/운동 환경이 중요하다.
- 20대 남자: 생활 인프라, 음식점, 카페 등 활동성 높은 지역을 보조 지표로 활용할 수 있다.
- 공원 많음: 공원 수, 1인당 공원면적, 공원율 등 park 관련 feature가 필요하다.
- 햄버거집 많음: 햄버거/버거/패스트푸드 관련 상권 feature가 필요하다.
- 동네 추천: 오늘 MVP에서는 서울 25개 자치구 단위로 추천한다.

## 2. 추천 단위

- 추천 단위는 **서울 25개 자치구**다.
- 공통 key는 `region_name`으로 통일한다.
- 행정동 단위 추천, 좌표 기반 거리 계산, 지도 히트맵은 오늘 범위에서 제외하고 추후 확장으로 둔다.
- 추천 로직은 원본 공공데이터 파일을 직접 읽지 않고, 전처리된 `processed/region_features.csv`만 참조한다.

## 3. 표준 Feature Table 개요

최종 표준 feature 파일은 다음 경로에 생성한다고 가정한다.

```text
processed/region_features.csv
```

이 파일은 원본 데이터와 추천 로직 사이의 중간 표준 테이블이다.

- 원본 데이터별 복잡한 컬럼명, 다중 헤더, 인코딩, 대용량 처리 문제는 `data_pipeline` 또는 `adapter` 계층에서 처리한다.
- 추천 로직은 `processed/region_features.csv`의 표준 컬럼만 사용한다.
- 실제 데이터에서 만들 수 있는 feature는 실제값으로 채운다.
- 오늘 직접 계산하기 어려운 feature는 현실성 있는 Mock 값으로 채운다.
- Mock으로 채우는 컬럼은 이 문서의 `실제값/Mock` 항목과 Mock 정책에 명확히 기록한다.
- Mock 여부가 추천 로직의 분기 조건이 되면 안 된다. 추천 로직은 동일한 컬럼 구조와 숫자 값만 보고 동작해야 한다.

## 5. Mock Feature 정책

오늘 Mock을 허용하는 feature는 다음이다.

- `park_count`
- `large_park_count`
- `running_friendly_score`
- `commercial_area_score`의 일부 보정값

Mock 정책은 다음 원칙을 따른다.

1. 서울 자치구별 특성을 어느 정도 반영한 현실적인 값으로 채운다.
   - 예: 한강공원, 올림픽공원, 서울숲, 북한산/불암산 인접 지역처럼 러닝/산책 이미지가 강한 지역은 상대적으로 높은 값을 줄 수 있다.
2. 실제 데이터로 교체 가능한 컬럼 구조를 유지한다.
   - Mock 컬럼도 실제 feature와 동일한 컬럼명, 타입, 값 범위를 사용한다.
3. 추천 로직이 Mock 여부에 의존하지 않게 한다.
   - 추천 로직은 `region_features.csv`의 숫자 값만 읽고 계산한다.
   - `is_mock` 여부는 문서 또는 별도 메타데이터에서만 관리한다.
4. Mock 컬럼은 문서에 명확히 기록한다.
   - 이 문서의 `실제값/Mock` 컬럼에 Mock 여부를 표시한다.
5. 추후 실제 데이터로 교체할 수 있어야 한다.
   - `park_count`, `large_park_count`: 개별 공원 목록/면적 데이터로 교체
   - `running_friendly_score`: 산책로, 체육시설, 보행환경, 야간 안전성 데이터로 교체
   - `commercial_area_score`: 상권 데이터의 업종별 집계와 인구/면적 대비 밀도로 교체

Mock 값은 오늘의 데모를 위한 임시값이므로 README, 발표자료, 데이터 문서에서 "일부 feature는 Mock 보완"이라고 명시한다.

## 7. region_features.csv 예시

아래는 `processed/region_features.csv`의 예시 형태다. 값은 스키마 검증과 추천 로직 설계를 위한 예시값이며, 실제 전처리 결과와 다를 수 있다.

```csv
region_name,park_count,park_area_per_person,park_ratio,large_park_count,food_count,cafe_count,hamburger_count,fastfood_count,running_friendly_score,commercial_area_score
송파구,35,18.5,24.8,3,18500,2100,42,95,0.88,0.82
마포구,28,9.4,12.6,1,22000,3400,55,120,0.72,0.91
관악구,31,15.2,21.3,2,14500,1500,33,80,0.79,0.66
강남구,26,8.7,10.4,1,31000,4200,65,140,0.63,0.95
노원구,42,20.1,29.5,3,12000,1100,24,60,0.86,0.58
```

예시 해석:

- `송파구`: 올림픽공원 등 대형 녹지 이미지가 강하므로 러닝 친화도 예시값을 높게 둔다.
- `마포구`: 카페/음식점/상권이 강하므로 `commercial_area_score`를 높게 둔다.
- `관악구`: 공원·산지 접근성이 있고 주거 밀도가 높은 편이므로 러닝 점수는 중상, 상권 점수는 중간으로 둔다.
- `강남구`: 상권은 매우 강하지만 공원/러닝 지표는 상대적으로 중간으로 둔다.
- `노원구`: 녹지와 산책 환경 이미지를 반영해 러닝 점수를 높게 둔다.

## 8. 오늘의 결론

오늘의 feature schema는 공원과 햄버거집 중심의 핵심 시나리오 성공을 위한 최소 구조다.

추천 단위는 서울 25개 자치구로 제한한다.

실제 데이터에서 만들 수 있는 feature는 실제값으로 사용하고, 러닝 친화도처럼 직접 계산이 어려운 feature는 현실성 있는 Mock으로 보완한다.

추천 로직은 원본 데이터 파일이 아니라 `processed/region_features.csv`만 참조하도록 설계한다.

주거비, 교통, 안전 feature는 이번 시나리오에서는 제외하고 추후 확장 feature로 관리한다.

따라서 다음 구현 단계에서는 원본 데이터별 전처리 코드를 `data_pipeline` 또는 `adapter` 계층에 두고, 최종적으로 `processed/region_features.csv`를 생성한 뒤 백엔드 추천 로직이 이 표준 feature table만 읽도록 변경한다.
